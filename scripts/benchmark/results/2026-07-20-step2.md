# Stage 2 performance at biobank scale

The main Stage 2 comparison is revision `8953759` against upstream REGENIE
v4.1.2 (`5f924b9`). At **N=500,000, M=700,000, and P=32 traits**, the batched
CPU implementation in `8953759` is 6.6-18.0x faster than upstream on the same
eight-core N2, depending on the model and missingness. The directly measured
complete quantitative run takes 1 hour 50 minutes with upstream and 8 minutes
18 seconds with `8953759`.

For a 100-million-variant scan, that changes the practical deployment from a
large and expensive upstream CPU fleet job to a much shorter 24-worker Spot
N2 job using `8953759`. One A100 running the CUDA backend at `8953759` is
faster than one N2 worker, but it is not fast or cheap enough to beat
chromosome-level CPU fan-out.

### Comparison map

| Question | Workload | Compared implementation | Baseline |
| --- | --- | --- | --- |
| Primary software speedup | N=500,000, M=700,000, P=32 on the same eight-core N2 | Batched CPU revision `8953759` | Upstream v4.1.2 (`5f924b9`) |
| CPU versus GPU hardware | N=500,000, M=700,000, P=32 | CUDA `8953759` on one A100 | CPU `8953759` on one eight-core N2 |
| Production placement | N=500,000, M=100,000,000, P=32 | CPU `8953759` on 24 chromosome workers | Upstream `5f924b9` on the same fleet, and CUDA `8953759` on one A100 |

Only the first and third rows include an upstream comparison. A statement
about A100 versus N2 is a same-revision hardware comparison, not evidence of
software speedup over upstream.

## Comparison with upstream v4.1.2

All CPU runs use the same real-LD-derived PGEN, oneMKL, eight physical cores,
`--bsize 1000`, `--ignore-pred`, and cold input. Each panel contains 32 traits
and one categorical covariate. Upstream is tag v4.1.2 at commit
`5f924b9bf54c1c7597174345def6eb2f1dee712c`.

| Model, N=500,000, M=700,000, P=32 | Missingness | Upstream `5f924b9` on N2 | Batched CPU `8953759` on N2 | `8953759` / upstream speedup | CUDA `8953759` on A100 |
| --- | --- | ---: | ---: | ---: | ---: |
| Quantitative | none | 1 h 49 m 44 s | 8 m 18 s | 13.23x | 3 m 59 s |
| Quantitative | 0-10% | 3 h 01 m 15 s* | 27 m 30 s | 6.59x | 12 m 24 s |
| Binary | none | 5 h 16 m 22 s* | 17 m 37 s | 17.96x | 5 m 03 s |
| Binary | 0-10% | 6 h 24 m 56 s* | 27 m 03 s | 14.23x | 5 m 40 s |
| Survival | none | 6 h 34 m 48 s* | 24 m 37 s | 16.03x | 11 m 58 s |
| Survival | 0-10% | 8 h 19 m 44 s* | 37 m 17 s | 13.41x | 13 m 06 s |

The complete quantitative upstream value is a direct 700,000-variant
measurement. Running every other upstream panel to completion would repeat an
already-flat per-block rate for hours, so the starred values scale complete
chromosome 1 runs from 58 to 711 blocks. Chromosome 1 contains 57,821 tested
variants in this fixture.

Two longer measurements check that projection. The chromosome-sized
quantitative rate predicts the completed upstream quantitative wall time
within 1.52%. A separate binary run processed 369 blocks before being stopped;
its wall time per completed block differs from the 58-block binary run by
0.22%.

The performance change preserves results. Across 192 matched chromosome 1
output files, all rows, labels, missing values, and tests agree. Of those
files, 146 are byte-identical. The remaining files differ only in printed
rounding: maximum absolute differences are 1e-8 for beta and standard error,
1e-6 for chi-square, and 1e-5 for `LOG10P`.

## Production comparison: N=500,000 and M=100,000,000

The production model assigns chromosomes 1-22, X, and Y to 24 co-located Spot
N2 workers. The N2 wall time below is the completion time of the slowest
chromosome. Its cost is the total for all 24 workers and their temporary
disks; it does **not** need to be multiplied by 24. The A100 column is one
on-demand `a2-highgpu-1g` running one REGENIE process over the whole scan.

| Model, N=500,000, M=100,000,000, P=32 | Missingness | Upstream `5f924b9`: 24-N2 fleet wall / total cost | Batched CPU `8953759`: 24-N2 fleet wall / total cost | CUDA `8953759`: one-A100 wall / total cost |
| --- | --- | ---: | ---: | ---: |
| Quantitative | none | 21.42 h / $96.34 | 1.93 h / $9.03 | 9.07 h / $44.94 |
| Quantitative | 0-10% | 35.15 h / $158.32 | 5.62 h / $25.64 | 28.94 h / $143.33 |
| Binary | none | 61.08 h / $274.60 | 3.71 h / $17.02 | 11.28 h / $55.87 |
| Binary | 0-10% | 74.24 h / $333.94 | 5.51 h / $25.15 | 12.60 h / $62.40 |
| Survival | none | 76.14 h / $342.21 | 5.01 h / $22.88 | 26.84 h / $132.92 |
| Survival | 0-10% | 96.28 h / $432.59 | 7.43 h / $33.78 | 29.46 h / $145.76 |

Against upstream `5f924b9`, the batched CPU implementation at `8953759`
reduces projected fleet wall time by
6.25-16.48x and total cost by 6.17-16.14x. This is the primary comparison. The
CUDA result is secondary: one A100 at `8953759` is much faster than upstream
CPU, but the `8953759` CPU implementation is better still once it is
distributed by chromosome.

The upstream production column uses the validated chromosome-sized rates.
Localization and result upload use the same model as the `8953759` CPU column.
The upstream disk estimate scales the `8953759` chromosome-weighted disk cost
with aggregate worker time; it is less precise than the compute estimate but
does not affect the placement decision.

### Placement assumptions

| Input or placement | Value |
| --- | --- |
| Samples | 500,000 |
| Stage 2 variants | 100,000,000 |
| Traits | 32 quantitative, binary, or survival traits |
| Shared work | Both placements start after the same A100 Stage 0/1 |
| CPU placement | 24 Spot `n2-standard-16` VMs, eight physical cores used, one VM per chromosome |
| GPU placement | One on-demand `a2-highgpu-1g`, all 12 host vCPUs, one REGENIE process |
| Data placement | Object storage and compute in the same region |
| Variant allocation | Proportional to GRCh38 chromosome length |
| Scratch space | Sized for each input and result with 10% headroom |

Prices captured on 2026-07-20 are $3.673385/hour for the A100 VM,
$0.254688/hour for a co-located Spot N2, $0.000136986/GiB-hour for balanced
Persistent Disk, and $0.000232877/GiB-hour for SSD Persistent Disk. The A100
VM is 14.42x the hourly price of one Spot N2 before disks.

## Why localization matters

The N=500,000 real-LD PGEN occupies 61.51 GB for 700,000 variants, or 87,868
bytes per variant. At the same density, 100 million variants occupy about
7.99 TiB.

A same-region copy of a 6.98 GiB PGEN to SSD Persistent Disk, including
`sync`, took 13.28 seconds: 0.526 GiB/s end to end.

| Placement | Data localized by the slowest unit | Localization wall |
| --- | ---: | ---: |
| Chromosome 1 on one N2 worker | 659.69 GiB | 20.9 min at the observed rate |
| All chromosomes on one A100 | 7.99 TiB | at least 174.6 min at the documented 800 MiB/s disk limit |

The N2 fleet transfers the same aggregate data, but each worker localizes only
one chromosome and the transfers can run in parallel. Chromosome 1 receives
8.06 million projected variants; chromosome 2 receives 7.84 million, while
chromosome 21 receives 1.51 million. The wall-time model uses chromosome 1,
not the 4.17-million-variant arithmetic mean.

The one-A100 placement needs roughly 9.12 TiB of scratch capacity after result
space and headroom. An `a2-highgpu-1g` can attach at most 2.93 TiB of Local
SSD, so the A100 model uses a capacity-appropriate balanced Persistent Disk.

## Direct `8953759` N2 and A100 measurements: N=500,000, M=700,000, P=32

These are direct cold-input measurements of revision `8953759`. N2 uses
oneMKL and eight physical cores. A100 uses CUDA, oneMKL, all 12 host vCPUs,
and one REGENIE process. This section compares hardware backends at the same
revision; it is not an upstream comparison.

| Model | Missingness | `8953759` N2 wall | `8953759` A100 wall | A100 speedup over N2 at `8953759` | Mean GPU utilization | Mean GPU power |
| --- | --- | ---: | ---: | ---: | ---: | ---: |
| Quantitative | none | 497.84 s | 238.65 s | 2.09x | 80.3% | 274 W |
| Quantitative | 0-10% | 1,650.04 s | 744.02 s | 2.22x | 91.1% | 333 W |
| Binary | none | 1,056.65 s | 303.47 s | 3.48x | 81.4% | 277 W |
| Binary | 0-10% | 1,622.77 s | 339.55 s | 4.78x | 82.6% | 261 W |
| Survival | none | 1,477.28 s | 718.44 s | 2.06x | 88.7% | 323 W |
| Survival | 0-10% | 2,236.61 s | 786.03 s | 2.85x | 88.7% | 307 W |

The GPU is not simply waiting idle: mean utilization is 80.3-91.1% and mean
power is 261-333 W. For complete quantitative traits, increasing `--bsize`
from 1,000 to 2,000 or 4,000 makes the cold A100 run 1.2% or 3.8% slower.
Giving the host side 12 threads instead of eight changes wall time by less
than one second. There is no obvious utilization or block-size adjustment
that closes the production cost and fan-out gap.

## Trait panels and scaling

The quantitative panel contains 32 correlated traits. Its missing-data
version increases per-trait missingness linearly from 0% to 10%.

Binary prevalences cycle through 1%, 2%, 5%, 10%, 20%, 30%, 40%, and 50%.
Survival event fractions cycle through 10% to 80% in ten-point increments.
The missing binary and survival panels reuse the quantitative missingness
masks.

The M=700,000 CPU runs also give a direct N=50,000 to N=500,000 comparison for
the `8953759` variant-compute phase, with P=32 in both sample-size panels:

| Model | Missingness | Variant-compute multiplier for 10x N |
| --- | --- | ---: |
| Quantitative | none | 8.54x |
| Quantitative | 0-10% | 9.32x |
| Binary | none | 9.21x |
| Binary | 0-10% | 9.32x |
| Survival | none | 9.30x |
| Survival | 0-10% | 9.34x |

Total wall time does not have one universal sample-size multiplier because
setup and output are mostly fixed while null fitting, genotype I/O, and score
calculation scale differently. The production model avoids that ambiguity by
using measurements taken directly at N=500,000.

These are score-only runs. Approximate Firth and other selected-test
corrections still run on the CPU. The available N=50,000, M=700,000, P=8
binary correction runs took 114.92 seconds on A100 at revision `ae9144d`,
144.31 seconds on N2 at revision `8953759`, and 219.41 seconds on T4 at
revision `ae9144d`. Because the N2 software revision differs, those values are
directional placement evidence rather than a controlled hardware speedup.
They are not upstream comparisons. Correction-heavy scans appear less
favorable to CUDA than the score-only table above, but a same-revision rerun
would be required for a precise ratio.

## Decision and limitations

Use the batched CPU implementation introduced in `8953759` on co-located Spot
N2 workers for production Stage 2. The CUDA backend measured at the same
revision is a valid single-process accelerator, but it is not competitive with
chromosome-level CPU fan-out on wall time or cost. T4 has no Stage 2 production
case in these measurements. CUDA should remain clearly experimental if there
is an active roadmap for another
multi-fold single-process improvement; otherwise, removing it would reduce
maintenance burden and user confusion.

The 100-million-variant values are projections, not whole-genome
measurements. They assume phase-linear variant scaling, sufficient aggregate
object-store bandwidth, same-region placement, and deletion of each worker
after its chromosome completes. They do not model VM startup, preemption,
different sex-chromosome tests, LOCO prediction parsing, or changes in the
rare-variant and correction mix. Long-lived source-data and result storage is
excluded because it is shared by the placements.
