# Stage 2 performance at biobank scale

The current CPU implementation is revision `c7e6c53`. On eight physical N2
cores it processes **781.2 variants/second** for **N=500,000, P=32
quantitative traits with 0-10% outcome missingness, and 700,000 variants
tested in Stage 2**. Upstream REGENIE v4.1.2 (`5f924b9`) processes an
estimated 64.4 variants/second, making the current code 12.14x faster for that
workload. For the matched **P=32 survival panel and 57,821 chromosome 1
variants**, the current code processes **435.7 variants/second**, 18.43x
upstream. Direct P=8 measurements show that trait-count scaling is
model-dependent: the speedups are 5.54x for quantitative traits, 11.44x for
binary traits, and 13.42x for survival traits.

Unless a table explicitly names a subphase, throughput means input variants
divided by end-to-end process wall time. It includes setup, null fitting,
genotype reading, association testing, and output for the complete joint
P-trait run. It is not the number of variant-trait tests per second; higher is
better.

This report reserves `M` for the number of markers used in Stage 1. Stage 2
tables spell out “variants tested” instead of reusing `M`. These score-only
benchmarks use `--ignore-pred`, so a Stage 1 marker count is not part of their
workload.

For a 100-million-variant scan, that changes the practical deployment from a
large and expensive upstream CPU fleet job to a much shorter 24-worker Spot
N2 job. The best measured branch CUDA result is faster than one N2 worker, but
it is not fast or cheap enough to beat chromosome-level CPU fan-out.

## Current CPU versus upstream

The current implementation removes a second sample-wide genotype pass for
traits with missing outcomes and parallelizes the several-GiB
genotype-squaring pass used by quantitative traits with missingness and by
binary traits. For survival models, it factors the common Cox projection out
of the phenotype-specific matrix multiplication and derives genotype norms
from hardcall counts collected during PGEN decoding. The table below uses the
same cold real-LD-derived chromosome 1 input for every row:
**N=500,000, P=32, and 57,821 variants tested in Stage 2**, with oneMKL and
eight physical N2 cores.

| Model (N=500,000; P=32; 57,821 Stage 2 variants tested) | Missingness | Upstream v4.1.2 (`5f924b9`), variants/s | Current CPU `c7e6c53`, variants/s | Current speedup |
| --- | --- | ---: | ---: | ---: |
| Quantitative | 0-10% | 65.2 | **739.8** | 11.35x |
| Binary | 0-10% | 30.7 | **684.1** | 22.29x |
| Survival | 0-10% | 23.6 | **435.7** | 18.43x |

All 96 output files behind the table passed byte-exact checks. These P=32
paths are unchanged by the quantitative dispatch patch, so their direct
measurements carry forward. For the survival run, factoring the projection
reduced the main crossproduct from
352 to 197 columns, while the hardcall norm handoff reduced the dense
squared-norm phase from 18.75 seconds to 0.20 seconds across 58 blocks.

Complete outcomes are faster for every model, but they do not change the
placement conclusion:

| Model (N=500,000; P=32; 57,821 Stage 2 variants tested) | Missingness | Upstream v4.1.2 (`5f924b9`), variants/s | Current CPU, variants/s | Current speedup |
| --- | --- | ---: | ---: | ---: |
| Quantitative | none | 106.0 | **1,317.1** | 12.42x |
| Binary | none | 37.3 | **597.6** | 16.00x |
| Survival | none | 29.9 | **587.0** | 19.62x |

The quantitative and binary rows carry forward direct measurements because
their code paths are unchanged. For complete survival, 16 of 32 files are
byte-identical to upstream, and only 26 of more
than 1.8 million result rows differ at all. Across those rows the largest
absolute differences are `1e-8` for beta, `1e-8` for chi-square, and `1e-6`
for `LOG10P`.

### Eight-trait workloads

The P=32 rates cannot simply be divided or multiplied to describe a smaller
joint analysis. The table below uses **N=500,000, P=8, 0-10% outcome
missingness, and 57,821 chromosome 1 variants tested in Stage 2**. The
hardware, oneMKL build, block size, and thread count are the same as in the
P=32 table. The refreshed quantitative input remained storage-backed but was
not forced cold; binary and survival retain their original cold-input
measurements.

| Model (N=500,000; P=8; 57,821 Stage 2 variants tested) | Upstream v4.1.2 (`5f924b9`), variants/s | Current CPU `c7e6c53`, variants/s | Current speedup |
| --- | ---: | ---: | ---: |
| Quantitative | 252.9 | **1,400.7** | 5.54x |
| Binary | 126.9 | **1,452.4** | 11.44x |
| Survival | 104.2* | **1,397.7** | 13.42x |

The quantitative row closes the former P=8 algorithm gap. Automatic dispatch
now selects block scoring for this N and P instead of the older per-variant
path. The direct chromosome-sized run completed in 41.28 seconds. It performed
about 3.70 GiB of physical filesystem reads and spent 13.38 seconds in
genotype input, versus 14.19 seconds for the prior cold run. The improvement
therefore comes from compute rather than a cached-input shortcut. All eight
files are byte-identical to upstream.

The binary runs completed: seven output
files are byte-identical, while the eighth differs only in the sixth printed
significant digit of `LOG10P`. The upstream survival value is a projection
from 32 of 58 blocks: after an 11.56-second fixed cost, the completed blocks
averaged 9.400 seconds per 1,000 variants. Across that measured prefix, all
eight survival outputs match current after numeric fields are rounded to five
significant digits, and three are byte-identical. The run was stopped once
the rate was stable rather than spending another four minutes confirming the
same steady state.

### Quantitative workload dispatch

Using the P=32 algorithm for every workload would regress important small
panels. The one-trait packed scorer is 1.54x faster than block scoring at
N=500,000, while the complete-trait per-variant path remains 6-14% faster in
several N=100,000-250,000 cells with P at or below 8. A cutoff based only on P
also leaves substantial performance unused at larger N.

The current CPU rule therefore uses both outcome missingness and workload
shape:

| Quantitative workload | Automatic CPU path |
| --- | --- |
| Complete outcomes, P=1 | Packed single-trait scorer |
| Complete outcomes, P>=12 | Block scoring |
| Complete outcomes, P>=4 and N × P>=2,000,000 | Block scoring |
| Other complete-outcome panels | Per-variant scoring |
| Phenotype-specific missingness, P>=2 | Block scoring |
| Single trait with missing outcomes | Existing strict single-trait path |

The crossover study used the same 12,000 real-LD-derived chromosome 1
variants at N=5,000, 50,000, 100,000, 250,000, and 500,000, with oneMKL and
eight physical cores. Complete-outcome block scoring is 1.14x, 1.41x, and
1.67x faster at N=500,000 for P=4, 8, and 12. With traits sampled across the
0-10% missingness range, it is 2.47x, 3.15x, 3.67x, and 4.80x faster for P=2,
4, 8, and 12. Representative cells on both sides of the automatic boundaries
were rerun through automatic dispatch; every result file was byte-identical to
the corresponding forced-path output.

The missing-outcome speedup uses more memory because it materializes the
squared genotype block once for the masked denominator crossproduct. At
N=500,000 and P=8, maximum RSS increased from 4,260,076 KiB to 8,307,588 KiB
on the 12,000-variant slice; the chromosome-sized run used 8,356,952 KiB.

The full quantitative run confirms that the chromosome-sized result carries
over to the complete 700,000-variant fixture:

| Model (N=500,000; P=32; 700,000 Stage 2 variants tested) | Missingness | Upstream v4.1.2 (`5f924b9`), variants/s | Current CPU `c7e6c53`, variants/s | Current speedup |
| --- | --- | ---: | ---: | ---: |
| Quantitative | 0-10% | 64.4* | **781.2** | 12.14x |

The current throughput comes from a direct cold-input measurement. The
starred upstream throughput scales its complete 57,821-variant chromosome 1
measurement to the 711 blocks in the full fixture; that method predicted
upstream's directly measured quantitative no-missingness throughput within
1.52%. All 32 current full-run output files passed exact-output regression
checks.

## How Stage 2 scales with sample count

A tenfold increase in N cannot be modeled as a tenfold increase in wall time.
The table pairs direct cold chromosome 1 runs at N=50,000 and N=500,000. Every
row has **P=32 inputs and 57,821 Stage 2 variants tested** on the same
eight-core N2 configuration.

| Model | Missingness | Upstream variants/s, N=50k -> N=500k | Upstream wall multiplier | Current variants/s, N=50k -> N=500k | Current wall multiplier | Current speedup, N=50k / N=500k |
| --- | --- | ---: | ---: | ---: | ---: | ---: |
| Quantitative | none | 1,578.5 -> 106.0 | 14.89x | 8,923.0 -> 1,317.1 | 6.77x | 5.65x / 12.42x |
| Quantitative | 0-10% | 844.3 -> 65.2 | 12.95x | 7,173.8 -> 739.8 | 9.70x | 8.50x / 11.35x |
| Binary | none | 651.1 -> 37.3 | 17.44x | 6,770.6 -> 597.6 | 11.33x | 10.40x / 16.00x |
| Binary | 0-10% | 482.8 -> 30.7 | 15.73x | 6,883.5 -> 684.1 | 10.06x | 14.26x / 22.29x |
| Survival | none | 459.3 -> 29.9 | 15.35x | 4,838.6 -> 587.0 | 8.24x | 10.53x / 19.62x |
| Survival* | 0-10% | 339.9 -> 23.6 | 14.38x* | 4,830.5 -> 435.7 | 11.09x* | 14.21x / 18.43x |

Upstream's measured N multiplier ranges from 12.95x to 17.44x across the five
fully comparable rows. Current ranges from 6.77x to 11.33x. The current
speedup generally widens at N=500,000 because the optimized sample-wide work
avoids the steeper upstream scaling.

For the starred N=50,000 survival-missingness row, Cox null fitting failed for
the same three traits in both implementations; each therefore analyzed 29 of
32 inputs. The N=500,000 runs analyzed all 32, so that row's N multiplier is
also affected by the analyzed trait count. It remains a valid same-N
upstream/current timing comparison, but not a clean sample-scaling estimate.

The N=50,000 outputs were compared directly. Across more than 10.9 million
comparable result rows, only ten printed rows differ at all. Nine differences
are at most `1e-9` in chi-square or `1e-11` in beta; the remaining row changes
chi-square from 5.15725 to 5.15724 while beta, standard error, and `LOG10P`
remain identical.

## Best measured CUDA comparison

The only full-fixture workload measured on both the current CPU code and the
best validated branch CUDA implementation is the quantitative missingness
panel. Both use the same cold real-LD-derived input and process all 32 traits
jointly.

| Model (N=500,000; P=32; 700,000 Stage 2 variants tested) | Missingness | Upstream v4.1.2, N2 variants/s | Current branch CPU, N2 variants/s | Best branch CUDA, A100 variants/s | CUDA util. / power |
| --- | --- | ---: | ---: | ---: | ---: |
| Quantitative | 0-10% | 64.4* | **781.2** | **940.8** | 91.1% / 333 W |

The A100 processes 1.20x as many variants per second as one N2 worker, but its
on-demand hourly price is 14.42x the price of one Spot N2 before disks. CUDA
is therefore not competitive with chromosome-level CPU fan-out. Earlier fork
CPU measurements are kept in the raw TSVs for reproducibility, not used as a
human-facing comparator.

## Production comparison: N=500,000 and 100,000,000 Stage 2 variants tested

This operational table retains elapsed wall time because completion time and
total cost, rather than aggregate fleet throughput, determine the placement.
The production model assigns chromosomes 1-22, X, and Y to 24 co-located Spot
N2 workers. The N2 wall time below is the completion time of the slowest
chromosome. Its cost is the total for all 24 workers and their temporary
disks; it does **not** need to be multiplied by 24. The A100 column is one
on-demand `a2-highgpu-1g` running one REGENIE process over the whole scan.

| Model (N=500,000; P=32; 100,000,000 Stage 2 variants tested) | Missingness | Upstream v4.1.2: 24-N2 fleet wall / total cost | Current branch CPU: 24-N2 fleet wall / total cost | Best branch CUDA: one-A100 wall / total cost |
| --- | --- | ---: | ---: | ---: |
| Quantitative | 0-10% | 35.15 h / $158.32 | **3.21 h / $14.79** | 28.94 h / $143.33 |

The current CPU projection is 10.95x faster and 10.70x cheaper than upstream.
It is also 9.01x faster and 9.69x cheaper than processing the whole scan on
one A100. The other model and missingness combinations have direct
chromosome-sized comparisons above, but are not promoted into whole-genome
cost projections without a full-fixture current phase anchor. Older fork
projections are not used as stand-ins.

The upstream production column uses the validated chromosome-sized rate. The
current CPU column uses the directly measured 700,000-variant quantitative
phase timings. All three placements include localization, result upload, and
temporary disks under the assumptions below.

### Placement assumptions

| Input or placement | Value |
| --- | --- |
| Samples | 500,000 |
| Stage 2 variants | 100,000,000 |

Prices captured on 2026-07-20 are $3.673385/hour for the A100 VM,
$0.254688/hour for a co-located Spot N2, $0.000136986/GiB-hour for balanced
Persistent Disk, and $0.000232877/GiB-hour for SSD Persistent Disk. The A100
VM is 14.42x the hourly price of one Spot N2 before disks.

## Why localization matters

The N=500,000 real-LD PGEN occupies 61.51 GB for 700,000 variants, or 87,868
bytes per variant. At the same density, 100 million variants occupy about
7.99 TiB.

A same-region copy of a 6.98 GiB PGEN to SSD Persistent Disk, including
`sync`, took 13.28 seconds: 0.526 GiB/s end to end.

| Placement | Data localized by the slowest unit | Localization wall |
| --- | ---: | ---: |
| Chromosome 1 on one N2 worker | 659.69 GiB | 20.9 min at the observed rate |
| All chromosomes on one A100 | 7.99 TiB | at least 174.6 min at the documented 800 MiB/s disk limit |

The N2 fleet transfers the same aggregate data, but each worker localizes only
one chromosome and the transfers can run in parallel. Chromosome 1 receives
8.06 million projected variants; chromosome 2 receives 7.84 million, while
chromosome 21 receives 1.51 million. The wall-time model uses chromosome 1,
not the 4.17-million-variant arithmetic mean.

The one-A100 placement needs roughly 9.12 TiB of scratch capacity after result
space and headroom. An `a2-highgpu-1g` can attach at most 2.93 TiB of Local
SSD, so the A100 model uses a capacity-appropriate balanced Persistent Disk.

## A100 diagnostics

The best measured A100 result and the current CPU result are in the single
N=500,000, P=32, 700,000-tested-variant table above. N2 uses oneMKL and eight
physical cores. A100 uses CUDA, oneMKL, all 12 host vCPUs, and one REGENIE
process.

The GPU is not simply waiting idle: mean utilization is 80.3-91.1% and mean
power is 261-333 W. For complete quantitative traits, increasing `--bsize`
from 1,000 to 2,000 or 4,000 makes the cold A100 run 1.2% or 3.8% slower.
Giving the host side 12 threads instead of eight changes wall time by less
than one second. There is no obvious utilization or block-size adjustment
that closes the production cost and fan-out gap.

## Trait panels and scaling

The quantitative panel contains 32 correlated traits. Its missing-data
version increases per-trait missingness linearly from 0% to 10%.

Binary prevalences cycle through 1%, 2%, 5%, 10%, 20%, 30%, 40%, and 50%.
Survival event fractions cycle through 10% to 80% in ten-point increments.
The missing binary and survival panels reuse the quantitative missingness
masks.

These are score-only runs. Approximate Firth and other selected-test
corrections still run on the CPU. Historical correction diagnostics remain in
the raw records but are not presented as branch comparisons because they have
not been repeated on the current revision and upstream v4.1.2.

## Decision and limitations

Use the current CPU implementation on co-located Spot N2 workers for
production Stage 2. The best measured CUDA backend is a valid single-process
accelerator, but it is not competitive with
chromosome-level CPU fan-out on wall time or cost. T4 has no Stage 2 production
case in these measurements. CUDA should remain clearly experimental if there
is an active roadmap for another
multi-fold single-process improvement; otherwise, removing it would reduce
maintenance burden and user confusion.

The 100-million-variant values are projections, not whole-genome
measurements. They assume phase-linear variant scaling, sufficient aggregate
object-store bandwidth, same-region placement, and deletion of each worker
after its chromosome completes. They do not model VM startup, preemption,
different sex-chromosome tests, LOCO prediction parsing, or changes in the
rare-variant and correction mix. Long-lived source-data and result storage is
excluded because it is shared by the placements.
