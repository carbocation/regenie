# Stage 2 benchmark: production placement and sample scaling

The current production choice is CPU Stage 2 on Spot instances. One A100 is
faster than one eight-core N2 worker for most score-only jobs, but it is not
fast enough to overcome the price difference or CPU fan-out. The measurements
also show that increasing the sample count from 50,000 to 500,000 is not a
simple 10x multiplier for end-to-end wall time.

## Production-shaped CPU estimate

This table estimates a **10-million-variant, 500,000-sample, 32-trait** scan on
8-core `n2-standard-16` Spot workers. Each cell is **wall time / total compute
cost across the entire worker fleet**. The cost has already been multiplied by
the worker count.

| Model | Missingness | 1 worker | 8 workers | 24 workers |
| --- | --- | ---: | ---: | ---: |
| Quantitative | none | 93.72 min / $0.523 | 11.79 min / $0.527 | 3.99 min / $0.534 |
| Quantitative | 0-10% | 395.09 min / $2.206 | 49.47 min / $2.210 | 16.55 min / $2.218 |
| Binary | none | 233.36 min / $1.303 | 29.33 min / $1.310 | 9.90 min / $1.327 |
| Binary | 0-10% | 386.73 min / $2.159 | 48.53 min / $2.168 | 16.33 min / $2.188 |
| Survival | none | 326.47 min / $1.823 | 41.20 min / $1.840 | 14.03 min / $1.880 |
| Survival | 0-10% | 546.07 min / $3.049 | 68.65 min / $3.066 | 23.18 min / $3.106 |

These are **score-only projections**, not measured 10-million-variant runs.
They scale the variant-dependent phases of measured N=500,000, M=16,000 runs
and repeat the fixed phases once per worker. The runs used `--ignore-pred`, so
LOCO prediction localization and parsing are not included. Genotypes were
already local; VM startup, data transfer, persistent disks, object-store
limits, output upload, Spot preemption, and association-test corrections are
also excluded. The table is useful for compute planning, not as a complete
cloud invoice. As a scale check, the per-variant rates inferred from the
N=50,000, M=16,000 runs agree with the measured M=700,000 rates within 2.5%
across all six panels; no equally large-M N=500,000 run has yet been made.

The near-constant cost across worker counts is expected. Equal-size shards
divide the same variant work among more workers, while repeated setup adds
only a small amount. In production, use enough workers to meet the wall-time
target after accounting for localization and scheduler overhead. At
N=500,000, increasing the scan from 10 million to 100 million variants would
make the compute portion almost 10x larger because the fixed phases are small.

## What 50,000 to 500,000 samples changes

The following runs directly compare the same oneMKL binary, hardware, 16,000
variants, 32-trait definitions, and missingness schedule. Both sample sizes use
`--ignore-pred`. The N=50,000 fixture has 10,000 samples from each of five
ancestry groups; the N=500,000 fixture uses every source sample and therefore
retains the source group proportions.

| Model | Missingness | N=50,000 wall | N=500,000 wall | Wall multiplier | Genotype + score multiplier |
| --- | --- | ---: | ---: | ---: | ---: |
| Quantitative | none | 2.94 s | 14.20 s | 4.83x | 7.93x |
| Quantitative | 0-10% | 5.63 s | 43.55 s | 7.74x | 9.88x |
| Binary | none | 4.73 s | 33.70 s | 7.12x | 9.41x |
| Binary | 0-10% | 6.15 s | 50.40 s | 8.20x | 9.82x |
| Survival | none | 7.33 s | 59.11 s | 8.06x | 9.22x |
| Survival | 0-10% | 9.12 s | 80.12 s | 8.79x | 10.38x |

Do not multiply the complete N=50,000 wall time by ten. On a small shard,
file initialization, phenotype loading, null-model fitting, and output are a
large fraction of the run and scale differently from the association scan.
For a large variant count, genotype processing and scoring dominate: the
measured multiplier is about 8x for complete quantitative traits and about
9-10x for the other panels. This is why the production estimate above is built
from fixed and per-variant phases separately.

Peak RSS at N=500,000 was 4.7 GB for complete quantitative traits, 9.0 GB for
missing quantitative traits, about 10 GB for binary traits, and 12.4-12.6 GB
for survival traits.

## Worker count is not chromosome count

There is no universally optimal worker count. A modern human whole-genome
scan has 24 chromosomes: 22 autosomes plus X and Y. That does not imply that
it should use exactly 24 workers.

The 24-worker rows above are a recognizable concurrency scenario. The 8-worker
case assigns an average of 1.25 million variants to each worker; the 24-worker
case assigns about 416,667. Both assume equal-variant shards and perfect
parallelism. A literal one-worker-per-chromosome schedule would be slower
because chromosome sizes are unequal; a practical scheduler can split large
chromosomes and combine small ones. The right worker count depends on the
wall-time target, localization bandwidth, shard balance, startup time, and
Spot reliability.

## Measured N=50,000, M=700,000 score performance

Every row is an end-to-end local run with 32 traits and real-LD-derived
genotypes. “A100 speedup” compares compute time with one eight-core N2 worker;
it does not include price or localization.

| Model | Missingness | N | M | 8-core N2 | A100 | A100 speedup | A100 mean GPU util. |
| --- | --- | ---: | ---: | ---: | ---: | ---: | ---: |
| Quantitative | none | 50,000 | 700,000 | 68.26 s | 37.68 s | 1.81x | 56.8% |
| Quantitative | 0-10% | 50,000 | 700,000 | 188.30 s | 78.56 s | 2.40x | 83.2% |
| Binary | none | 50,000 | 700,000 | 109.52 s | 34.18 s | 3.20x | 66.7% |
| Binary | 0-10% | 50,000 | 700,000 | 168.75 s | 36.04 s | 4.68x | 70.6% |
| Survival | none | 50,000 | 700,000 | 154.27 s | 65.37 s | 2.36x | 79.4% |
| Survival | 0-10% | 50,000 | 700,000 | 228.50 s | 65.99 s | 3.46x | 80.9% |

The A100 is 1.81-4.68x faster than one N2 worker in these runs. That is below
the 10.97x A100-to-Spot-N2 hourly price ratio. High GPU utilization in the
wider panels does not change the placement decision.

## CPU implementation result

Revision `8953759` prepares phenotype and null-model terms once and scores a
full genotype block with oneMKL matrix multiplication. It is used for
quantitative panels with at least 16 traits and binary or survival panels with
at least four traits. Smaller quantitative panels retain the packed
per-variant path.

| Model | Missingness | Previous CPU | Current CPU | Speedup |
| --- | --- | ---: | ---: | ---: |
| Quantitative | none | 92.36 s | 68.26 s | 1.35x |
| Quantitative | 0-10% | 263.40 s | 188.30 s | 1.40x |
| Binary | none | 723.67 s | 109.52 s | 6.61x |
| Binary | 0-10% | 810.59 s | 168.75 s | 4.80x |
| Survival | none | 1,013.75 s | 154.27 s | 6.57x |
| Survival | 0-10% | 1,028.59 s | 228.50 s | 4.50x |

The packed hardcall decoder also reduced the 32-trait missing-data genotype
phase to 68.01 seconds for quantitative traits, 68.89 seconds for binary
traits, and 84.67 seconds for survival traits. Output comparisons are
byte-identical for that decoder-only change.

## CUDA scope and conclusion

The CUDA backend keeps residuals, masks, weights, and projection terms
resident for a chromosome and accepts packed PGEN hardcalls. Filtering,
formatting, SPA/Firth corrections, and exact Cox scoring remain on the CPU.
Unsupported workflows use the CPU implementation.

One REGENIE process is the supported baseline. The T4 does not provide useful
end-to-end price/performance, and no further T4 Stage 2 work is justified by
the current results. A100 Stage 2 CUDA is useful as a local accelerator when
data and the GPU are already resident, but it is not a production placement
argument. If that does not change materially, keeping CPU as the documented
production path will avoid overstating the value of Stage 2 CUDA.

## Validation and raw data

- Block scoring has deterministic tests for complete and missing
  quantitative, binary, and Cox scores.
- Across the six 32-trait comparisons, beta differences are at most `1e-7`
  and the largest printed chi-square or log-p difference is about `1e-6`;
  most files are byte-identical.
- The one-trait N=500,000 control is byte-identical and remains on the packed
  path.
- CPU-only oneMKL and A100 CUDA+oneMKL builds pass the Step 1, Step 2, and Cox
  Firth compute tests.
