# Stage 2 benchmark: CPU fan-out is the production default

This report describes the current Stage 2 implementation as of 2026-07-20.
The headline measurements use **50,000 samples and 700,000 variants** with
realistic LD. The production projection keeps **N=50,000** and scales the
measured variant-dependent phases to **10 million variants**. It is not a
500,000-sample measurement.

## Bottom line

- On one eight-core `n2-standard-16`, blockwise oneMKL scoring cuts the
  32-trait workloads by 1.35-6.61x relative to the previous CPU code.
- One A100 is still 1.81-4.68x faster than one N2 worker for the measured
  32-trait score-only runs. That is below the 4.73x on-demand price ratio and
  far below the 10.97x A100-to-Spot-N2 price ratio.
- For a projected 10-million-variant scan, an ideal 22-worker Spot N2 fan-out
  is 4.06-8.60x faster and 2.03-4.29x cheaper than keeping one A100.
- Approximate Firth corrections weaken the GPU case further because the
  selected corrections still run on the CPU.
- Stage 2 CUDA remains useful when the data and A100 are already resident. It
  is not currently a reason to place a production Stage 2 scan on a GPU VM.

## Benchmark definition

| Item | Value |
| --- | --- |
| Measured samples (`N`) | 50,000 |
| Measured variants (`M`) | 700,000 |
| Genotypes | Hard-call PGEN derived from real genetic data; local LD and allele-frequency structure retained |
| Quantitative panel | 32 traits, complete or per-trait missingness increasing from 0% to 10% |
| Binary panels | 8 or 32 traits; prevalences cycle through 1%, 2%, 5%, 10%, 20%, 30%, 40%, and 50% |
| Survival panels | 8 or 32 models; event fractions cycle from 10% through 80% |
| CPU | 8 physical Xeon cores, 8 REGENIE threads, oneMKL 2026.1 |
| A100 | `a2-highgpu-1g`, one REGENIE process, 12 host threads, oneMKL 2026.1 |
| Block size | 1,000 variants |
| Prediction mode | LOCO for 8-trait and quantitative runs; `--ignore-pred` for synthetic 32-trait binary/survival panels |
| Cache/localization | Genotypes already local; repeated trait runs did not force a cold page cache |

The complete and missing panels use the same outcomes. The missing binary and
survival panels inherit the quantitative missingness masks, averaging 5%
missingness. Fixture generation is deterministic, and independently generated
files matched by SHA-256 across systems.

Spot break-even.

**N=50,000**, and **M=700,000**.

prediction mode.

and output by `10,000,000 / 700,000`.

The N2 wall time assumes 22 equal variant shards with perfect parallelism.
Each worker pays the fixed phases. Neither architecture includes VM startup,

The projection understates the operational advantage of CPU fan-out. A
whole-GWAS A100 would need the complete 10-100M-variant dataset localized to
one machine. N2 workers can localize disjoint chromosome chunks in parallel
and use smaller disks. The useful worker count will ultimately depend on
transfer bandwidth, startup time, shard balance, and Spot reliability.

This projection still uses N=50,000. Runtime at N=500,000 must be measured; it
should not be obtained by treating the table above as a production forecast.

## CPU implementation result

Revision `8953759` prepares phenotype and null-model terms once per applicable
workflow and scores a full genotype block with matrix multiplication. It is
used for quantitative panels with at least 16 traits and binary or survival
panels with at least four traits. Smaller quantitative panels retain the packed
per-variant path, which is faster and uses less memory for one trait.

| Model | Traits | Missingness | N | M | Previous CPU | Current CPU | Speedup |
| --- | ---: | --- | ---: | ---: | ---: | ---: | ---: |
| Quantitative | 32 | none | 50,000 | 700,000 | 92.36 s | 68.26 s | 1.35x |
| Quantitative | 32 | 0-10% | 50,000 | 700,000 | 263.40 s | 188.30 s | 1.40x |
| Binary | 32 | none | 50,000 | 700,000 | 723.67 s | 109.52 s | 6.61x |
| Binary | 32 | 0-10% | 50,000 | 700,000 | 810.59 s | 168.75 s | 4.80x |
| Survival | 32 | none | 50,000 | 700,000 | 1,013.75 s | 154.27 s | 6.57x |
| Survival | 32 | 0-10% | 50,000 | 700,000 | 1,028.59 s | 228.50 s | 4.50x |
| Binary, approximate Firth | 8 | none | 50,000 | 700,000 | 252.47 s | 144.31 s | 1.75x |
| Survival, approximate Firth | 8 | none | 50,000 | 700,000 | 313.70 s | 132.35 s | 2.37x |

Missing-trait PGEN input also uses the packed hardcall decoder before the
existing phenotype-specific allele-count bookkeeping. This reduces the
32-trait genotype phase from 93.26 to 68.01 seconds for quantitative traits,
94.92 to 68.89 seconds for binary traits, and 107.95 to 84.67 seconds for
survival traits. The decoder-only output comparisons are byte-identical.

## CUDA scope

The CUDA backend accepts packed PGEN hardcalls directly and keeps residuals,
masks, weights, and projection terms resident for a chromosome. It supports
additive single-variant hardcall tests with separate phenotype outputs, all
analyzed samples present in the genotype input, and at most 16 covariates.
Filtering, formatting, SPA/Firth corrections, and exact Cox scoring remain on
the CPU. Unsupported workflows use the CPU implementation.

In CUDA builds, `--compute-backend auto` uses CUDA for supported binary and
survival scores and keeps quantitative Stage 2 on the CPU for pre-Ampere GPUs.
Explicit `--compute-backend cuda` is useful for controlled measurement. A
single REGENIE process is the supported baseline; earlier multi-process A100
experiments were diagnostics for missing internal overlap, not a recommended
deployment shape.

The T4 reaches high utilization in some kernels but does not deliver useful
end-to-end price/performance. No further T4 Stage 2 optimization is justified
by the present measurements.

## Numerical validation

- The block backend has deterministic tests for complete and missing
  quantitative scores, binary scores, and Cox scores.
- Across the six full 32-trait comparisons, output structure is unchanged,
  beta differences are at most `1e-7`, and the largest printed chi-square or
  log-p difference is about `1e-6`; most files are byte-identical.
- The one-trait, 500,000-sample control is byte-identical and stays on the
  packed path.
- The full binary Firth output is byte-identical. Seven of eight survival
  Firth files are byte-identical; one chi-square changes from `0.988900` to
  `0.988899`.
- CPU-only oneMKL and A100 CUDA+oneMKL builds pass `step1_compute_cpu`,
  `step1_compute_auto`, `step2_compute_cpu`, and `cox_firth`.
