# Step 2 performance checkpoint: 2026-07-20

Step 2 is no longer simply a variant-reading problem for multi-trait work. On
an eight-core N2, the retained CPU changes make a 32-trait quantitative run
with variable missingness 2.97x as fast, an eight-trait binary run 1.22x as
fast, and an eight-trait survival run 1.38x as fast. These are end-to-end
measurements over 700,000 real-LD-derived variants.

A separate packed-input CUDA probe also found enough arithmetic headroom to
justify an integrated implementation. In particular, its projected scoring
time over 700,000 variants is 29-43 seconds on T4 and 3.7-4.5 seconds on A100
for eight binary- or survival-shaped traits. Those are deliberately reported
as kernel-level bounds: the probe does not read a PGEN, apply corrections, or
write association results.

## Workload

The CPU runs use `n2-standard-16-smt-off`: eight physical Xeon cores, eight
visible CPUs, 62 GiB of RAM, and oneMKL 2026.1. Every run uses eight REGENIE
threads, a block size of 1,000, 50,000 samples, and the same 700,000-variant
PGEN derived from 1000 Genomes haplotypes. This is the same physical fixture
used for the earlier cross-system work, not an independent-marker substitute.

The phenotype panels are:

- 32 quantitative traits with per-trait missingness increasing from 0% to
  10%, for 80,000 missing values in total;
- eight binary traits with prevalences from 1% to 50%; and
- eight time-to-event traits with observed-event fractions from 10% to 80%.

The binary and survival headline runs use approximate Firth correction. A
16,000-variant pilot also measures score-only and corrected modes separately.
All binaries in the comparison link to oneMKL.

## Retained CPU result

The baseline is `114ef81`; the optimized implementation is `db1ce23`.

| Model | Traits | Missingness or correction | Baseline wall | Optimized wall | Change | Throughput |
| --- | ---: | --- | ---: | ---: | ---: | ---: |
| Quantitative | 32 | 0-10% by trait | 783.62 s | 263.47 s | -66.4% | 2.97x |
| Binary | 8 | approximate Firth | 307.40 s | 252.56 s | -17.8% | 1.22x |
| Survival | 8 | approximate Firth | 433.29 s | 313.80 s | -27.6% | 1.38x |

The quantitative missingness result is the largest change. Genotype work falls
from 415.53 to 93.14 seconds, while variant compute falls from 345.13 to
147.54 seconds. The optimized full run spends 16.22 seconds reading LOCO
predictions and 4.04 seconds writing results, leaving 21 seconds outside the
two main stages.

Binary genotype work is essentially unchanged at 22.79 seconds. Its compute
stage falls from 274.28 to 219.61 seconds. Survival compute falls from 395.78
to 276.51 seconds; genotype work remains 23.01 seconds. These two results show
that variant reading is not the dominant cost for the non-Gaussian panels.

The retained implementation makes three changes:

- Missing-phenotype accounting updates only the traits absent for a sample.
  Quantitative sparse and dense score paths likewise adjust only missing
  traits instead of repeatedly applying full sample-by-trait masks.
- Dense binary scoring uses cached small projection crossproducts. It only
  materializes a sample-length residualized genotype when Firth or SPA is
  actually selected for that test.
- The default Cox score and projected norm use cached Gram and score terms.
  A sample-length residualized genotype is deferred until a selected Firth
  correction needs it. `--coxscore-exact` keeps the original calculation.

Complete-case quantitative performance is unchanged in the pilot: 5.59
seconds before and 5.55 seconds after. This is an important control because
the new missingness bookkeeping should not tax the already-fast complete-case
path.

The two full-run differences are consistent with the changed floating-point
reduction order in algebraically equivalent formulas. No corrected test
failed in either full non-Gaussian run. The optimized build also passes the
CPU backend, automatic-backend, and Cox/Firth test targets.

## CUDA feasibility

`bfff7de` adds `step2_packed_score_cuda.cu`, a standalone probe for the data
shape an integrated backend would receive. It keeps phenotype residuals,
masks, weights, and projection terms resident on the GPU, then measures a
1,000-variant packed-hardcall upload and one of three scoring kernels. The
reported 700,000-variant figure is 700 times the measured upload-plus-kernel
latency.

### Thirty-two quantitative-shaped traits

| GPU system | Upload per block | Kernel per block | Projected 700k scoring | Mean GPU util. | Median util. | Mean power |
| --- | ---: | ---: | ---: | ---: | ---: | ---: |
| `n1-standard-8-t4` | 2.700 ms | 48.811 ms | 36.06 s | 96.0% | 100% | 65.8 W / 70 W |
| `a2-highgpu-1g` | 2.564 ms | 3.956 ms | 4.56 s | 93.0% | 100% | 303.6 W / 400 W |

### Eight binary- and survival-shaped traits

| GPU system | Kernel | Upload per block | Kernel per block | Projected 700k scoring | Mean GPU util. | Mean power |
| --- | --- | ---: | ---: | ---: | ---: | ---: |
| `n1-standard-8-t4` | weighted projection | 2.554 ms | 38.452 ms | 28.70 s | 92.9% | 65.2 W / 70 W |
| `n1-standard-8-t4` | Cox dual projection | 2.554 ms | 58.862 ms | 42.99 s | 92.9% | 65.2 W / 70 W |
| `a2-highgpu-1g` | weighted projection | 2.530 ms | 2.794 ms | 3.73 s | 82.6% | 250.7 W / 400 W |
| `a2-highgpu-1g` | Cox dual projection | 2.530 ms | 3.900 ms | 4.50 s | 82.6% | 250.7 W / 400 W |

The T4 is at 93-94% of its power limit and has a 100% median utilization in
both sustained runs. The A100 also has a 100% median, but its eight-trait run
averages 83% utilization and 63% of its power limit because the smaller grid
does not occupy it as continuously. This is useful headroom, but it is not by
itself a performance result.

The probe intentionally excludes PGEN metadata and filtering, allele-frequency
and missing-genotype handling, sparse/dense path selection, Firth and SPA,
result formatting, and disk output. It also uses generated packed bytes rather
than the real-LD fixture. LD does not change the amount of dense score
arithmetic, but the probe cannot establish end-to-end elapsed time or numerical
equivalence. Its projections also stand in for the small terms needed by the
real null models rather than reproducing every REGENIE branch.

The conclusion is therefore narrower than “Step 2 takes four seconds on an
A100.” The measurements show that packed decoding and multi-trait score
crossproducts can saturate both target GPUs at rates far above the current CPU
compute stage. That is enough to make an integrated implementation worth
building. T4 remains a plausible distributed fan-out target; A100 has enough
headroom to be considered for a fast single-machine Step 1 plus Step 2 run.

## Integration target

The next implementation should read a normal PGEN block once, transfer its
packed hardcalls directly, and retain all chromosome-static null-model terms
on the device. Double-buffered streams can overlap the next PGEN read and
upload with the current kernel and result download. The GPU should return
small per-variant, per-trait score terms; CPU code should continue to own
filtering, output, and selective Firth/SPA correction. A sample-length genotype
should only be expanded for a test that actually needs a correction.

The first acceptance test is not GPU utilization. It is lower cold-cache wall
time than `db1ce23`, with matching results, on the same real-LD-derived input.
Both `n1-standard-8-t4` and `a2-highgpu-1g` need that end-to-end test before a
CUDA Step 2 path should be retained.
